# SecureZef Engine 🛡️ (Native Go)

SecureZef is an ultra-lightweight, blazing-fast Go wrapper that intercepts output streams of any AI agent (Python, Node, Bash, Claude Code). It scans streams in real-time against strict DLP (Data Loss Prevention) and Prompt Injection policies before the output hits the terminal or the user.

Forget bloated API proxies. This is 100% Go. It's invisible, highly configurable, and blocks leaks instantly.

---

## 🧠 How It Works

Unlike traditional API gateways that introduce network latency, SecureZef operates at the **process level**:
1. You run your agent *through* SecureZef.
2. SecureZef attaches to the agent's `stdout` and `stderr` pipes.
3. Every chunk of data generated by the LLM is scanned in memory using pre-compiled regex engines.
4. If a rule is violated, SecureZef instantly emits an `[ALERT]` and can terminate or sanitize the output.

---

## 🚀 Quick Start

### 1. Build the Engine
```bash
go build -o securezef main.go
```

### 2. Run your agent securely
Wrap your existing AI script (`agent.py`, `node app.js`, etc.):
```bash
./securezef -rules rules/master_rules_v1.json -script agent.py
```

## 📜 Managing Rules & Policies

Rules are defined in a simple JSON file: `rules/master_rules_v1.json`. You don't need to recompile the Go binary to add new rules—just update the JSON and restart the wrapper.

### Adding Custom Rules
To add a new rule, append a JSON object to the `rules` array.

```json
{
  "id": "KEY_LEAK_OPENAI",
  "category": "dlp_key_leak",
  "pattern": "sk-proj-[a-zA-Z0-9_-]+",
  "severity": "critical",
  "description": "Blocks OpenAI token leaks"
}
```

### Directory-Specific Blocking (Feature)
You can configure rules to block the AI from interacting with specific sensitive directories. For example, to prevent an agent from reading or deleting anything in `/root/` or `~/.ssh/`:

```json
{
  "id": "DIR_PROTECTION_SSH",
  "category": "file_system_protection",
  "pattern": "(?i)(cat|rm|ls|vi|nano|echo)\\s+(~|/home/[^/]+)?/\\.ssh/.*",
  "severity": "critical",
  "description": "Blocks any read/write access to SSH directories"
}
```

## 🔌 API & Stream Interfaces

SecureZef intercepts raw IO streams. Here is the data flow:
- **Input (STDIN):** Passed directly to the child process (the Agent).
- **Output (STDOUT/STDERR):** Scanned in chunks.
- **Alerts:** Emitted to `STDERR` using a structured format for easy parsing by parent applications.

**Alert Format:**
```text
[ALERT] <STREAM_NAME> match '<Matched_Text>' (Rule: <Rule_Description>, Sev: <Severity>)
```
*Future Roadmap: A structured JSON HTTP API endpoint is planned for v2.1 to allow remote observability.*

## 🤖 Integration as a Skill

SecureZef can be natively integrated into your favorite AI environments.

### As an OpenClaw Skill
You can enforce SecureZef globally in OpenClaw by updating your tool configuration to prefix all shell executions with the binary.
1. Compile `securezef` and place it in your `$PATH`.
2. Update OpenClaw's `exec` tool configuration to alias the shell through `securezef`:
   `"command": "securezef -rules /path/to/rules.json -script bash -c '...'"`

### As a Claude Code Integration
When running Anthropic's Claude Code CLI, wrap the binary to ensure Claude doesn't leak local environment variables or touch restricted directories:
```bash
alias claude="securezef -rules ~/.securezef/rules.json -script claude"
```
Now, whenever you type `claude`, it runs completely sandboxed by Go heuristics.

## 🛠 Project Structure
- `main.go`: The CLI entry point and process wrapper (intercepts IO pipes).
- `engine/engine.go`: The core regex scanning engine. Compiles rules on startup for zero-latency streaming.
- `rules/master_rules_v1.json`: The active ruleset repository.
